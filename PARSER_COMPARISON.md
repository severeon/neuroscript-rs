# Parser Library Comparison for NeuroScript v2

## Overview

This document compares four Rust parser libraries for NeuroScript v2, with concrete examples of parsing our syntax.

---

## Test Case: Parse Neuron Definition

```neuroscript
neuron Linear(in_features: Nat, out_features: Nat):
  in: [*, in_features]
  out: [*, out_features]
  impl: core,nn/Linear
```

---

## 1. pest (PEG Parser Generator)

### Grammar File (`grammar.pest`)

```pest
WHITESPACE = _{ " " | "\t" }
COMMENT = _{ "#" ~ (!NEWLINE ~ ANY)* }

program = { SOI ~ neuron_def+ ~ EOI }

neuron_def = {
    "neuron" ~ ident ~ params ~ ":" ~ NEWLINE
    ~ INDENT ~ port_decls ~ body ~ DEDENT
}

params = { "(" ~ param ~ ("," ~ param)* ~ ")" }
param = { ident ~ ":" ~ type_expr }

port_decls = { port_decl+ }
port_decl = { ("in" | "out") ~ ":" ~ shape ~ NEWLINE }

body = { impl_ref | graph }
impl_ref = { "impl:" ~ impl_path ~ NEWLINE }

impl_path = { ident ~ "," ~ ident ~ "/" ~ ident }
shape = { "[" ~ shape_elem ~ ("," ~ shape_elem)* ~ "]" }
shape_elem = { wildcard | ident }
wildcard = { "*" }

ident = @{ ASCII_ALPHA ~ (ASCII_ALPHANUMERIC | "_")* }
type_expr = { "Nat" | "Float" | "String" }

INDENT = { "  " }  // Simplified for example
DEDENT = { "" }
NEWLINE = _{ "\n" | "\r\n" }
```

### Pros
- ✅ **Declarative grammar** - Easy to read and modify
- ✅ **Built-in error reporting** - Good error messages out of the box
- ✅ **PEG semantics** - No ambiguity, handles left-recursion
- ✅ **Active development** - Well-maintained, good docs
- ✅ **Indentation support** - Can handle via custom rules

### Cons
- ❌ **Runtime parsing** - Grammar not compiled (slower than lalrpop)
- ❌ **Manual AST construction** - Need to walk parse tree yourself
- ❌ **Indentation is tricky** - Not first-class like Python's parser

### Verdict: ⭐⭐⭐⭐⭐ **Best choice**
- Best balance of maintainability and features
- Excellent for domain-specific languages
- **Recommended for NeuroScript v2**

## 2. lalrpop (LALR Parser Generator)

### Grammar File (`grammar.lalrpop`)

```lalrpop
use crate::ast::*;

grammar;

pub Program: Program = {
    <neurons:NeuronDef+> => Program { neurons }
};

NeuronDef: NeuronDef = {
    "neuron" <name:Ident> <params:Params> ":" "\n"
    INDENT <ports:PortDecl+> <body:Body> DEDENT => {
        NeuronDef { name, params, ports, body }
    }
};

Params: Vec<Param> = {
    "(" <Comma<Param>> ")"
};

Param: Param = {
    <name:Ident> ":" <ty:Type> => Param { name, ty }
};

PortDecl: PortDecl = {
    <dir:("in" | "out")> ":" <shape:Shape> "\n" => {
        PortDecl { direction: dir, shape }
    }
};

Body: Body = {
    "impl:" <ImplPath> => Body::Impl(<>),
    "graph:" "\n" <Graph> => Body::Graph(<>),
};

ImplPath: String = {
    <provider:Ident> "," <lib:Ident> "/" <name:Ident> => {
        format!("{},{}/{}", provider, lib, name)
    }
};

Shape: Shape = {
    "[" <Comma<ShapeElem>> "]" => Shape(<>)
};

ShapeElem: ShapeElem = {
    "*" => ShapeElem::Wildcard,
    <Ident> => ShapeElem::Var(<>),
};

Type: Type = {
    "Nat" => Type::Nat,
    "Float" => Type::Float,
};

Ident: String = r"[a-zA-Z_][a-zA-Z0-9_]*" => <>.to_string();

Comma<T>: Vec<T> = {
    <mut v:(<T> ",")*> <e:T?> => {
        if let Some(e) = e {
            v.push(e);
        }
        v
    }
};

// Note: INDENT/DEDENT would need custom lexer
match {
    r"\s+" => { },  // Skip whitespace
    r"#[^\n]*" => { },  // Skip comments
    _
}
```

### Parser Code

```rust
// Auto-generated by lalrpop
lalrpop_mod!(pub grammar);

fn parse_neuron(input: &str) -> Result<NeuronDef, ParseError> {
    grammar::NeuronDefParser::new().parse(input)
}
```

### Pros
- ✅ **Compile-time generation** - Fast parsing (grammar compiled)
- ✅ **Direct AST construction** - Types match your AST directly
- ✅ **Type-safe** - Rust types in grammar
- ✅ **Good error recovery** - Can suggest fixes

### Cons
- ❌ **Harder to debug** - LALR conflicts cryptic
- ❌ **Manual lexer for indentation** - Need custom tokenizer
- ❌ **Steeper learning curve** - LALR theory needed
- ❌ **Less flexible** - Some grammars don't fit LALR

### Verdict: ⭐⭐⭐⭐ **Good alternative**
- Faster than pest but harder to maintain
- Consider if performance is critical
- **Not recommended due to indentation complexity**

## 3. nom (Parser Combinators)

### Parser Code

```rust
use nom::{
    IResult,
    branch::alt,
    bytes::complete::{tag, take_while1},
    character::complete::{char, multispace0, alpha1, alphanumeric1},
    combinator::{map, recognize},
    multi::{many0, separated_list0},
    sequence::{delimited, preceded, separated_pair, tuple},
};

fn ident(input: &str) -> IResult<&str, String> {
    map(
        recognize(tuple((
            alpha1,
            many0(alt((alphanumeric1, tag("_"))))
        ))),
        |s: &str| s.to_string()
    )(input)
}

fn param(input: &str) -> IResult<&str, Param> {
    map(
        separated_pair(
            ident,
            delimited(multispace0, char(':'), multispace0),
            type_expr
        ),
        |(name, ty)| Param { name, ty }
    )(input)
}

fn params(input: &str) -> IResult<&str, Vec<Param>> {
    delimited(
        char('('),
        separated_list0(
            delimited(multispace0, char(','), multispace0),
            param
        ),
        char(')')
    )(input)
}

fn shape_elem(input: &str) -> IResult<&str, ShapeElem> {
    alt((
        map(char('*'), |_| ShapeElem::Wildcard),
        map(ident, ShapeElem::Var)
    ))(input)
}

fn shape(input: &str) -> IResult<&str, Shape> {
    map(
        delimited(
            char('['),
            separated_list0(
                delimited(multispace0, char(','), multispace0),
                shape_elem
            ),
            char(']')
        ),
        Shape
    )(input)
}

fn port_decl(input: &str) -> IResult<&str, PortDecl> {
    map(
        tuple((
            alt((tag("in"), tag("out"))),
            preceded(
                delimited(multispace0, char(':'), multispace0),
                shape
            )
        )),
        |(dir, shape)| PortDecl {
            direction: dir.to_string(),
            shape
        }
    )(input)
}

fn impl_ref(input: &str) -> IResult<&str, String> {
    preceded(
        tag("impl:"),
        preceded(
            multispace0,
            take_while1(|c: char| c.is_alphanumeric() || c == ',' || c == '/')
        )
    )(input).map(|(i, s)| (i, s.to_string()))
}

fn neuron_def(input: &str) -> IResult<&str, NeuronDef> {
    map(
        tuple((
            preceded(tag("neuron"), preceded(multispace0, ident)),
            params,
            preceded(multispace0, char(':')),
            // ... would need complex indentation handling here
        )),
        |(name, params, _, _)| NeuronDef {
            name,
            params,
            ports: vec![],
            body: NeuronBody::Primitive("".to_string())
        }
    )(input)
}
```

### Pros
- ✅ **Maximum flexibility** - Can parse anything
- ✅ **Zero-copy** - Efficient, no allocations
- ✅ **Composable** - Build complex parsers from simple ones
- ✅ **Excellent error types** - Detailed failure info

### Cons
- ❌ **Verbose** - Much more code than declarative
- ❌ **Manual error messages** - No automatic error reporting
- ❌ **Harder to read** - Parser is code, not grammar
- ❌ **Indentation very manual** - Need full lexer with state

### Verdict: ⭐⭐⭐ **Too low-level**
- Powerful but too much boilerplate
- Error messages require significant work
- **Not recommended** - Use for low-level parsing only

## 4. tree-sitter (Incremental GLR Parser)

### Grammar File (`grammar.js`)

```javascript
module.exports = grammar({
  name: 'neuroscript',

  extras: $ => [/\s/, $.comment],

  rules: {
    program: $ => repeat1($.neuron_def),

    neuron_def: $ => seq(
      'neuron',
      field('name', $.identifier),
      field('params', $.params),
      ':',
      $._newline,
      $._indent,
      repeat1($.port_decl),
      $.body,
      $._dedent
    ),

    params: $ => seq(
      '(',
      commaSep($.param),
      ')'
    ),

    param: $ => seq(
      field('name', $.identifier),
      ':',
      field('type', $.type)
    ),

    port_decl: $ => seq(
      choice('in', 'out'),
      ':',
      field('shape', $.shape),
      $._newline
    ),

    body: $ => choice(
      $.impl_ref,
      $.graph
    ),

    impl_ref: $ => seq(
      'impl:',
      $.impl_path,
      $._newline
    ),

    impl_path: $ => seq(
      $.identifier,
      ',',
      $.identifier,
      '/',
      $.identifier
    ),

    shape: $ => seq(
      '[',
      commaSep($.shape_elem),
      ']'
    ),

    shape_elem: $ => choice(
      '*',
      $.identifier
    ),

    type: $ => choice('Nat', 'Float', 'String'),

    identifier: $ => /[a-zA-Z_][a-zA-Z0-9_]*/,

    comment: $ => /#[^\n]*/,

    _newline: $ => '\n',
    _indent: $ => token('  '),  // Simplified
    _dedent: $ => token('')
  }
});

function commaSep(rule) {
  return optional(seq(rule, repeat(seq(',', rule))))
}
```

### Parser Code (Rust Bindings)

```rust
use tree_sitter::{Parser, Language};

extern "C" { fn tree_sitter_neuroscript() -> Language; }

fn parse_neuron(source: &str) -> Result<NeuronDef, Error> {
    let mut parser = Parser::new();
    let language = unsafe { tree_sitter_neuroscript() };
    parser.set_language(language)?;

    let tree = parser.parse(source, None).unwrap();
    let root = tree.root_node();

    // Walk tree and construct AST
    let mut cursor = root.walk();
    for child in root.children(&mut cursor) {
        match child.kind() {
            "neuron_def" => {
                // Parse neuron definition
                let name_node = child.child_by_field_name("name").unwrap();
                let name = source[name_node.byte_range()].to_string();
                // ... etc
            }
            _ => {}
        }
    }

    Ok(NeuronDef { /* ... */ })
}
```

### Pros
- ✅ **Incremental parsing** - Reparse only changed parts (LSP)
- ✅ **Error resilient** - Partial trees on errors
- ✅ **Multiple bindings** - Rust, Python, JS, etc.
- ✅ **Great for editors** - Built for IDE features
- ✅ **First-class indentation** - Native support

### Cons
- ❌ **C dependency** - Requires C compiler
- ❌ **Grammar in JavaScript** - Not pure Rust
- ❌ **Overkill for compiler** - Designed for editors
- ❌ **More complex setup** - Need tree-sitter CLI

### Verdict: ⭐⭐⭐⭐ **Future tooling**
- Perfect for LSP server (Phase 7)
- Too heavyweight for initial compiler
- **Use later for IDE support, not MVP**

## Recommendation: pest

### Why pest wins for NeuroScript v2

1. **Best maintainability**: Grammar is clear and easy to modify
2. **Good enough performance**: Runtime parsing is fine for compiler
3. **Excellent errors**: Built-in error reporting with miette integration
4. **Active community**: Well-documented, many examples
5. **Pure Rust**: No external dependencies
6. **PEG semantics**: No ambiguity, predictable behavior

### Migration path

**Phase 1** (MVP): Use pest for compiler
```
Source → pest → HIR → MIR → PyTorch
```

**Phase 7** (Tooling): Add tree-sitter for LSP
```
Editor → tree-sitter → Syntax highlighting + completion
Compiler still uses pest
```

This gives us:
- Fast iteration during development (pest)
- Production compiler with good errors (pest)
- Amazing editor support later (tree-sitter)

## Conclusion

**Use pest for NeuroScript v2 compiler**
- Declarative, maintainable, good errors
- Sufficient performance for our needs
- Easy to extend as language grows

**Consider tree-sitter for LSP in Phase 7**
- Incremental parsing for real-time feedback
- Error resilience for incomplete code
- Industry standard for editor tooling

This gives us the best of both worlds: fast development now, excellent tooling later.
